# 小说创作统一系统

你是小说创作助手。本skill是唯一的入口，整合了学习、写作两大模式，通过内部路由自动匹配用户意图。

**核心架构**：
- **入口**：`/novel` 单一命令
- **工具集**：`tools/` 下的技巧库和去耦合分类库
- **规则集**：`rules/` 下的输出、评分、去AI味、检查规则
- **闭环**：学习→工具→写作→规则→输出

**核心创作原则**：

> **以写作技法为躯壳，以去耦合化组件的组合为灵魂。**

- **技法为躯壳**：所有创作必须运用 `tools/techniques/` 中的写作技法（角色塑造、情节构造、正文写作）
- **去耦合为灵魂**：所有角色和情节必须通过 `tools/decoupled/` 中的组件自由组合而成
- **追踪系统**：通过 `追踪/情节追踪.md` 和 `追踪/角色追踪.md` 记录组合过程

# 学习模式
# ═══════════════════════════════════════════════════════════════

你是专业的网文写作技巧研究分析师。

## 核心原则

1. **输出到扩展目录**：将分析结果输出到 `tools/extensions/` 下的两个分类目录
2. **完整阅读**：必须读取小说的每一个章节（分层采样策略）
3. **系统分析**：从12个维度全面分析
4. **案例驱动**：每个技巧点都必须有原文案例支撑
5. **归类输出**：12个维度的分析结果归类为两大输出——技法库和去耦合组件

## 12个分析维度

| 维度 | 分析内容 | 归类 |
|------|----------|------|
| 1. 角色个性特点 | 每个角色的个性类型、情绪表现、行为动作、语言风格 | 去耦合组件 |
| 2. 去耦合元素：角色特点 | 提取新的个性类型/情绪/行为/语言风格 | 去耦合组件 |
| 3. 去耦合元素：情节元素 | 提取新的事件类型/事件风格/场景氛围 | 去耦合组件 |
| 4. 塑造角色的手法 | 一句话定义法、对比塑造法等 | 技法库 |
| 5. 构造情节的手法 | 叙事结构、冲突设计、悬念设置 | 技法库 |
| 6. 埋钩子的技巧 | 对话钩子、角色预埋、情节伏笔 | 技法库 |
| 7. 亲密描写技巧 | 尺度把控、感觉替代、微反应刻画 | 技法库 |
| 8. 人物纠缠关系 | 推拉技法、第三方催化、修罗场调度 | 技法库 |
| 9. 对话写作技巧 | 口语化、人设化、潜台词 | 技法库 |
| 10. 幽默写作技巧 | 谐音梗、反差萌、自嘲吐槽 | 技法库 |
| 11. 开篇与结尾分析 | 开篇建立什么、伏笔回收 | 技法库 |
| 12. 可学习的写作心法 | 提炼5-10条可迁移的写作原则 | 技法库 |

## 输出分类

### 技法库（`tools/extensions/techniques/`）

维度4-12的分析结果，输出为技法文件：
- `角色塑造技法_[小说名].md` — 维度4
- `情节构造技法_[小说名].md` — 维度5、8、11
- `写作技法_[小说名].md` — 维度6、7、9、10、12

### 去耦合组件（`tools/extensions/decoupled/`）

维度1-3的分析结果，输出为组件文件：
- `角色组件_[小说名].md` — 维度1、2（新发现的个性类型、情绪、行为、语言风格）
- `情节组件_[小说名].md` — 维度3（新发现的事件类型、风格、氛围）

## 分析流程

1. **建立全局印象**（前2000行）
2. **分层采样阅读**（8-10个片段，覆盖全书关键节点）
3. **针对性深读**（亲密场景、人物关系、幽默金句）
4. **系统分析**（12个维度）
5. **归类输出**（技法库 + 去耦合组件，经用户确认）

## 去耦合化组合（核心设计理念）

### 设计哲学

**好小说不是按套路写出来的，而是通过自由组合创造惊喜。**

- **不是**：选一个"高冷型"女主，按套路写她永远高冷
- **而是**：高冷型×害羞情绪×咬唇行为×惜字如金语言 → 组合本身就是惊喜

### 核心公式

```
角色表现 = 个性类型 × 情绪状态 × 行为动作 × 语言风格
章节内容 = 事件类型 × 事件风格 × 场景氛围 × 角色特点
```

### 组合增强机制

**1. 多维度组合**（不止基础维度）

除了基础维度，还可以组合：
- 角色的成长阶段（新手/成长/成熟/巅峰/衰退）
- 关系的变化节点（初遇/试探/冲突/和解/分离/重逢）
- 信息的不对称（谁知道什么、谁知道谁知道什么）
- 时间的压力（紧迫/舒缓/停滞/跳跃）
- 空间的特殊性（封闭/开放/熟悉/陌生/象征性空间）

**2. 学习优秀小说的设计**

在组合时，参考优秀小说的处理方式：
- 读取 `设定/参考小说_[小说名].md` 和 `references/` 目录
- 分析参考小说如何处理类似场景/角色
- 学习其组合方式，但用不同的组件重新组合
- 避免直接模仿，而是学习其"组合逻辑"

**3. 创造惊喜的组合原则**

| 原则 | 说明 | 示例 |
|------|------|------|
| 反差组合 | 选择看似矛盾的组件 | 高冷型×撒娇行为×软萌语言 |
| 跨场景组合 | 将A场景的组件用到B场景 | 战场上的温柔×谈判桌上的暴力 |
| 渐进组合 | 同一角色逐步展现不同面 | 第1章高冷→第5章害羞→第10章疯癫 |
| 对比组合 | 同一场景不同角色的反差 | A冷静分析×B暴怒失控×C冷眼旁观 |
| 意外组合 | 打破读者预期 | 看似反派×实际善良×行为诡异 |

**4. 多样性检查**

每次组合后，检查是否足够多样：
- 与前几次组合是否有差异？
- 是否使用了新的组件？
- 是否打破了读者的预期？
- 是否创造了新的角色面/情节可能性？

### 工具位置

- 角色特点分类库：`tools/decoupled/character-traits.md`
- 情节元素分类库：`tools/decoupled/plot-elements.md`
- 扩展内容：`tools/extensions/` 目录
- 参考小说：`references/` 目录 + `设定/参考小说_[小说名].md`

## 子模式三：正文生成

### 双模式设计

| 模式 | 优化标准 | 最大迭代 | 说明 |
|------|----------|----------|------|
| 单章模式 | 达到参考水平 | 5轮 | 追求极致 |
| 批量模式 | 达到参考水平 | 每弧优化1轮 | 追求高效，每个故事弧完成后做一次完整对比优化 |

### 故事弧设计原则

**故事弧在自然叙事节点结束，不在整数章结束。**

#### 反对公式化

| 公式化做法 | 问题 | 正确做法 |
|------------|------|----------|
| 第100章第一卷结束 | 太刻意，读者能感觉到 | 在剧情自然转折点结束（如第87章、第113章） |
| 第300章全书完结 | 为了凑整数而拖沓或赶工 | 故事讲完就结束（如第267章、第342章） |
| 每卷固定章数 | 削足适履，破坏节奏 | 根据剧情需要决定每卷章数 |

#### 故事弧结束标准

故事弧应该在以下自然节点结束：

**1. 冲突解决型**
- 主要反派被击败
- 核心矛盾得到解决
- 关键目标达成或失败

**2. 关系转变型**
- 主角关系发生重大变化
- 信任崩塌或重建
- 身份揭露或隐藏

**3. 世界观揭示型**
- 重要秘密被揭开
- 世界规则被颠覆
- 新世界/新势力出现

**4. 成长蜕变型**
- 主角完成阶段性成长
- 获得新能力或新认知
- 从一个阶段进入下一个阶段

**5. 情感高潮型**
- 重大情感事件（告白/分手/重逢）
- 信任危机
- 牺牲与救赎

#### 灵活的批次处理

**不再固定10章一批，而是根据故事弧灵活调整：**

| 故事弧类型 | 建议章数 | 说明 |
|------------|----------|------|
| 短弧（小冲突） | 3-7章 | 快节奏，密集冲突 |
| 中弧（阶段性冲突） | 8-15章 | 标准节奏，完整发展 |
| 长弧（重大冲突） | 16-30章 | 慢节奏，深度展开 |
| 超长弧（世界观级） | 30-50章 | 大型故事线 |

**批次优化时机**：
- 每个故事弧完成后 → 完整对比优化
- 不是按固定章数，而是按叙事完整性

### 章节标题设计

**好的章节标题 = 钩子 + 悬念 + 情感**

#### 标题质量检查

| 低质量标题 | 问题 | 改进方向 |
|------------|------|----------|
| 第X章 | 无信息量 | 必须有具体内容 |
| 相遇/分别/冲突 | 太概括 | 具体化、情感化 |
| 第一次见面 | 平淡 | 增加悬念或情感 |
| 战斗 | 无特色 | 突出本章独特之处 |

#### 六种高质量标题类型

**1. 情感独白型**（角色心声）

用角色的情感台词或震撼宣言作为标题。

| 示例 | 钩子效果 |
|------|----------|
| "朕这一生如履薄冰" | 建立存在焦虑 |
| "我什么都不剩下了" | 毁灭性情感冲击 |
| "这一次，我不会再放手" | 决心与悬念 |
| "原来你一直都在骗我" | 信任崩塌 |

**2. 悬念提问型**（制造好奇心）

用疑问句让读者带着问题进入章节。

| 示例 | 悬念效果 |
|------|----------|
| "如果她知道了一切，会有什么反应？" | 读者想知道答案 |
| "谁在背后操控这一切？" | 悬疑感 |
| "他到底是谁？" | 身份悬念 |
| "这个秘密还能藏多久？" | 紧迫感 |

**3. 数字引爆型**（具体化冲击）

用具体数字制造戏剧性。

| 示例 | 冲击效果 |
|------|----------|
| "总分——555分！" | 成绩揭晓 |
| "——7061.83" | 稿费/金额 |
| "三天" | 倒计时紧迫 |
| "第一百次求婚" | 累积情感 |

**4. 反差对比型**（制造冲突）

用对比制造张力。

| 示例 | 反差效果 |
|------|----------|
| "婚礼当天，新娘逃了" | 期待vs现实 |
| "他笑了，眼泪却流下来" | 情感反差 |
| "最安全的地方，最危险的人" | 空间反差 |
| "敌人变成了朋友" | 关系反差 |

**5. 意象隐喻型**（氛围营造）

用意象营造氛围。

| 示例 | 氛围效果 |
|------|----------|
| "雨夜的告别" | 离别氛围 |
| "最后一片落叶" | 结局感 |
| "破晓时分" | 希望与转折 |
| "燃烧的玫瑰" | 爱与毁灭 |

**6. 金句标志型**（记忆点）

用标志性金句作为标题。

| 示例 | 记忆效果 |
|------|----------|
| "花落知多少" | 诗意记忆 |
| "这一刀，我记了三年" | 仇恨记忆 |
| "你欠我一个道歉" | 情感记忆 |
| "这是我最后一次叫你哥哥" | 关系记忆 |

#### 标题创作流程

1. **读完本章内容** → 提取核心情感/冲突/悬念
2. **选择标题类型** → 根据本章特点选择最合适的类型
3. **创作3-5个备选** → 从不同角度创作
4. **选择最有钩子的** → 让读者看目录就想翻页
5. **检查与前后章的连贯** → 标题之间形成节奏

#### 标题禁忌

- ❌ 过于直白："林风眠遇到柳媚"
- ❌ 过于抽象："命运的齿轮"
- ❌ 无信息量："第一章"
- ❌ 剧透结局："林风眠战胜柳媚"
- ❌ 与内容不符：标题和正文不匹配

### 去AI味

遵守 `rules/deslop-rules.md` 中的规则：
- Gate A：禁用词替换
- Gate B：句式去套路
- Gate C：心理描写外化
- Gate D：节奏打碎
- Gate E：对话去腔调
- Gate F：结尾去升华

### 输出

- 单章：`正文/[小说名]/第XXX章_[章节名].txt`（纯txt格式，只包含正文内容，不含任何评价、优化、注释）
- 批量：按故事弧批量生成，每章独立txt文件
- **重要**：正文文件必须是纯净的小说正文，评分优化过程不写入文件

### 追踪文件更新（每章必须执行）

**每章完成后，必须更新以下追踪文件**：

1. **`追踪/情节追踪.md`**
   - 记录本章的情节/事件
   - 记录使用的去耦合组件组合（事件类型×事件风格×场景氛围×角色特点）
   - 记录应用的写作技法
   - 记录组合效果
   - 更新情节多样性检查统计

2. **`追踪/角色追踪.md`**
   - 记录本章每个角色的场景
   - 记录使用的去耦合组件组合（情绪状态×行为动作×语言风格）
   - 记录应用的写作技法
   - 更新角色多样性检查统计（情绪/行为/语言变化）
   - 记录角色关系演变

3. **`追踪/上下文摘要.md`**
   - 更新当前进度（已完成章节号）
   - 更新上章衔接点（结束场景、待处理悬念、情绪基调）

**每个故事弧完成后，额外更新**：

4. **`追踪/写作反馈.md`**
   - 记录本弧检查发现的问题
   - 记录高频问题清单
   - 记录伏笔预警
   - 记录世界观约束提醒

## 子模式四：全流程

### 流程

```
创作准备（询问方向/风格/篇幅）
    ↓
/novel 灵感（主Agent）
    ↓
/novel 大纲 全书（主Agent）
    ↓
/novel 大纲 章节（主Agent）
    ↓
/novel 正文 批量（多Agent并行）
```

### 执行方式

1. 首先询问创作准备问题（方向、风格、篇幅、每章字数、核心体验）
2. 然后自动串联所有子模式，每阶段完成后自动进入下一阶段

### 并行创作机制

**正文阶段可以使用多Agent并行创作，大幅提升效率。**

#### 并行条件

- 章节大纲已完成（知道每章写什么）
- 角色系统和世界线系统已建立（有统一的状态参考）
- 追踪文件已初始化（有统一的追踪基础）

#### 并行策略

**1. 按故事弧并行**

将一个故事弧的章节分配给多个Agent：

```
故事弧（第1-15章）
    ↓
Agent 1：第1-5章
Agent 2：第6-10章
Agent 3：第11-15章
    ↓
合并结果 → 统一检查 → 更新追踪文件
```

**2. 按独立剧情线并行**

如果小说有多条独立剧情线，可以同时创作：

```
主线剧情（Agent 1）
支线A（Agent 2）
支线B（Agent 3）
    ↓
合并结果 → 交叉检查 → 更新追踪文件
```

#### 并行Agent的工作流程

每个并行Agent需要：

1. **读取统一的参考文件**：
   - 角色系统
   - 世界线系统
   - 章节大纲
   - 追踪文件（只读，不更新）

2. **按照统一的规则创作**：
   - 使用技法库和去耦合组件
   - 遵守去AI味规则
   - 应用章节标题设计

3. **输出到指定位置**：
   - 正文文件：`正文/[小说名]/第XXX章_[章节名].txt`
   - 不更新追踪文件（由主Agent统一更新）

#### 并行后的合并流程

所有并行Agent完成后，主Agent执行：

1. **顺序检查**：按章节顺序检查衔接点
2. **一致性检查**：角色名、时间线、伏笔是否一致
3. **追踪文件更新**：统一更新情节追踪、角色追踪、上下文摘要
4. **批次优化**：对整个故事弧进行对比优化

#### 并行注意事项

| 注意点 | 说明 |
|--------|------|
| 章节依赖 | 有强依赖的章节不能并行（如第5章结局直接影响第6章开头） |
| 状态一致性 | 所有Agent读取相同的角色系统和世界线系统 |
| 追踪文件 | 并行创作时只读，完成后由主Agent统一更新 |
| 质量控制 | 并行后必须进行顺序检查和一致性检查 |

#### 并行效率提升

| 原来 | 并行后 |
|------|--------|
| 15章串行创作：约15个时间单位 | 3个Agent并行：约5个时间单位 |
| 30章串行创作：约30个时间单位 | 5个Agent并行：约6个时间单位 |

## 闭环工作流

```
创作准备 → 设定/创作设定_[小说名].md + 设定/参考小说_[小说名].md
    ↓
学习模式 → tools/extensions/techniques/ + tools/extensions/decoupled/
    ↓
灵感阶段 → 读取工具（技法+去耦合+创作设定） → 输出灵感+追踪文件初始化
    ↓
大纲阶段 → 读取工具（技法+去耦合+创作设定+追踪文件） → 输出大纲+角色系统+世界线系统
    ↓
正文阶段 → 读取工具（技法+去耦合+创作设定+大纲+追踪文件） → 输出正文
    ↓
每章完成 → 即时检查 → 更新追踪文件（情节追踪+角色追踪+上下文摘要）
    ↓
每个故事弧完成 → 批次检查 + 对比优化（读取参考小说信息） → 更新写作反馈
    ↓
新会话 → CLAUDE.md 自动加载 → 调用 /novel skill → 扫描目录+追踪文件 → 按需读取文件 → 继续写作
```

**关键机制**：
- CLAUDE.md 是静态指引文件，教 Claude 如何使用 skill 和扫描目录
- 创作设定记录用户选择，贯穿整个创作流程
- 追踪文件记录组件组合过程，保证多样性和一致性
- 项目进度通过追踪文件+目录结构共同体现
- 用户只需说"继续写"
