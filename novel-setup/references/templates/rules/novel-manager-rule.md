# Novel Manager Rules

小说项目管理与一致性规范。当用户编辑设定、追踪或大纲文件时自动加载。

本规则合并了原 novel-manage-rule 和 novel-consistency-rule，并新增自动管理协议。

---

## 自动管理协议

### 核心机制

系统通过 PostToolUse hook 自动检测内容变更，触发 novel-supervisor agent 进行审查。

**自动触发条件**：
- Write/Edit 工具修改了 `正文/` 或 `设定/` 目录下的文件
- hook 输出指令，Claude 调用 novel-supervisor 执行审查

**审查内容**：
1. 5维度一致性检查（调用 novel-checker）
2. 追踪文件自动更新（进度/伏笔/时间线）
3. 写作反馈生成（追踪/写作反馈.md）

### 手动触发

用户也可以手动触发管理操作：
- `/novel-setup 检查` — 全书一致性检查
- `/novel-setup 检查章 [文件]` — 单章检查
- `/novel-setup 进度 更新` — 手动更新进度

## 世界观变更规则

修改 `设定/世界观.md` 时必须：
1. **检查伏笔引用**：该设定是否被伏笔引用？修改是否影响伏笔回收？
2. **检查正文使用**：Grep 正文确认无矛盾
3. **检查时间线影响**：如有影响，更新 `追踪/时间线.md`
4. **记录变更日志**：在世界观文档末尾的变更日志中记录

### 更新时机
| 触发事件 | 更新内容 | 更新章节 |
|----------|----------|----------|
| 新角色登场 | 添加角色速查条目 | 第七节 |
| 角色状态变化 | 更新当前状态 | 第七节 |
| 新势力出现 | 添加势力条目 | 第三节 |
| 势力关系变化 | 更新关系描述 | 第三节 |
| 新地点出现 | 添加地点条目 | 第四节 |
| 重要事件发生 | 添加事件记录 | 第五节 |
| 新伏笔埋设 | 添加伏笔条目 | 第八节 |
| 伏笔回收 | 更新伏笔状态 | 第八节 |
| 新规则建立 | 添加规则条目 | 第六节 |
| 获得新能力 | 添加能力条目 | 第二节 |

## 角色设定规则

### 新角色必须创建独立文件

每个新角色必须在 `设定/角色/` 创建独立文件，包含：
- **基本信息**：姓名、年龄、身份
- **外貌描述**：2-3 个标志性特征
- **性格特征**：至少 1 个反差点
- **语言风格**：口头禅/标志性句式
- **与主角的关系**

### 修改角色设定时必须

1. Grep 正文中该角色的所有出场
2. 确认修改不与已有描写矛盾
3. 如有矛盾，标记需要修改的正文位置

---

## 一致性检查规则

### 设定变更必须检查关联文件

修改任何设定文件时，必须检查：
1. **伏笔文件**：该设定是否被伏笔引用？修改是否影响伏笔回收？
2. **时间线文件**：该设定是否影响时间线？
3. **角色文件**：该设定是否影响角色能力/状态？

### 时间变更必须同步

故事内时间变化时，必须同步更新：
1. `追踪/时间线.md`（如存在）
2. 相关角色的年龄（如果跨越生日）
3. 季节/天气描述的一致性

### Grep 检查要求

修改设定后，使用 Grep 搜索正文中是否有与新设定矛盾的已有描写：
- 搜索角色名 + 关键属性
- 搜索地点名 + 关键特征
- 搜索力量体系相关术语

## 伏笔追踪规则

### 新增伏笔时
1. 在 `追踪/伏笔.md` 记录：埋设位置、计划回收位置、关联角色
2. 确保伏笔有明确的回收计划
3. 更新世界观文档第八节

### 回收伏笔时
1. 更新 `追踪/伏笔.md` 的回收状态
2. 确保回收方式与埋设一致
3. 更新世界观文档第八节

### 超期处理
- 超过预计回收章节 20 章以上标记为"超期"
- 超期伏笔需优先处理
- 可调整回收计划或在当前章节回收

---

## 进度更新规则

### 自动更新（由 novel-supervisor 执行）
- 每章正文完成后自动更新
- 每批次完成后自动更新

### 手动更新
- 用户运行 `/novel-setup 进度 更新`

### 更新内容
1. 更新 `追踪/进度.md` 的当前章节和已完成数
2. 更新字数统计
3. 如有世界观变更，更新相关设定文件

### 进度预警
- 连续 3 章评分低于 8.0：提示质量下降
- 超过 10 章未回收伏笔：提示伏笔超期
- 进度停滞超过 3 天：提示继续创作

## 严重程度定义

| 级别 | 标记 | 定义 | 处理方式 |
|------|------|------|----------|
| **硬伤** | 🔴 | 直接矛盾、逻辑不通、违反已建立规则 | 必须修复后才能继续 |
| **中等** | 🟡 | 不够合理、可能引起读者困惑 | 建议修复，可后续修正 |
| **轻微** | 🟢 | 细节瑕疵、不影响大局 | 可选修复，记录在案 |

### 处理流程
```
发现问题
    ├─ 硬伤 → 立即修复 → 重新检查 → 确认通过 → 继续创作
    ├─ 中等 → 记录问题 → 建议修复方案 → 用户决定修复时机
    └─ 轻微 → 记录问题 → 供用户参考
```
